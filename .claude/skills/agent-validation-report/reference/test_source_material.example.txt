【專案會議記錄 2026/07/22】預算AI審核 Agent 上線前驗證會議
出席:張佳雯(廠務部門,本次報告人)、廠務系統窗口 x2

一、專案背景
這個 Agent 的目的是對廠務系統所有預算案件做 AI 初步預審,提供廠務專家一個參考,
減少人工審核的 cycle time。目前現況是廠務人員收到 PBMS 系統 FAC 關卡通知後,
要人工下載案件、用 Excel 記錄、依經驗判斷歸屬、派發給對應專家,平均一件要 4 小時。

Agent 的角色定位是「預算初審助理」— 負責產出批准/退件建議,並附上類似的歷史案例引用給專家參考,
最終核准權還是在廠務專家手上,Agent 不做最終決策。

觸發時機:PBMS 系統的 FAC 關卡收到新的預算申請時,系統會發 Email 通知,Agent 就開始跑。

成功門檻(這次會議討論定案):AI 預審結論要跟廠務專家最終審核的結果一致率達到 85% 以上才算過關。
專案指標的部分,目標是把 cycle time 從現況 4 小時/件降到 2 小時以內,也就是要降低超過 50%。

二、資料與模型
資料來源主要是 PBMS 系統的歷史預算案件,還有廠務工程系統的歸屬對照表(哪個案件類型該分給哪個系統的專家)。
模型是用公司內部 ProphetAI 的線上 agent,審核這個任務有自己獨立的 agent id 跟金鑰,
跟流程圖生成、申訴合理性判讀是分開的三個 agent,不共用。
知識庫的部分是把過去核准/退件的案例做成 RAG,每一輪專家複審完的結果都會自動寫回去更新,
讓系統越用越準。

三、決策邏輯
Sub-Agent 叫做「審核 Agent」,負責的任務是初審、派發、跟複審回饋這三塊。
思考邏輯是:先用 RAG 判斷這個案件屬於哪個廠務系統,信心分數要 ≥0.7 才會自動分類,
低於這個門檻就轉人工判斷,不會硬猜。用到的工具是 ProphetAI 的審核 agent 加上 RAG 檢索。

四、Guardrail 設計(這次會議定案 5 條紅線)
1. Output(輸出格式):AI 一定要回固定的 JSON 格式,欄位如果缺漏就視為無法審核,不能亂填。
2. Capability(能力邊界):如果 API 逾時或連不上,系統要標示「無法審核」,絕對不能捏造一個
   審核結論出來充數。
3. Grounding(依據原則):AI 給的審核建議一定要引用申請文件裡實際寫的內容,不能無中生有掰數字。
4. Hallucination(幻覺防護):如果 AI 的結論跟 PBMS 系統的實際數據矛盾(比如AI說沒問題但系統
   顯示金額異常),要強制轉人工複審,不能自己蓋過去。
5. Goal(目標邊界):要一直提醒大家 AI 只是輔助建議,最終核准權還是在廠務專家身上,
   不能讓 AI 自動核准取代人工決策。

五、黃金測試案例討論(這次先定案 3 個情境,第 4 個下次會議再補)
情境 1 — 案件缺少必要附件:
  問題狀況:申請人沒附上報價單就送出申請。
  專家標準決策:退件並具體註明缺哪些附件。
  Agent 驗證檢查點:Agent 的退件說明裡要列出缺失的必要文件項目,不能只說「資料不全」帶過。

情境 2 — 申請理由寫得很模糊:
  問題狀況:申請理由只寫「例行採購」,沒有具體用途說明。
  專家標準決策:退回,要求補充具體用途。
  Agent 驗證檢查點:Agent 要判定為「待補件」,不能直接核准通過。

情境 3 — 申請金額異常:
  問題狀況:申請金額比歷史同類案件平均高出 3 倍以上。
  專家標準決策:轉人工複審,不能自動核准。
  Agent 驗證檢查點:這個情況要觸發 Hallucination 紅線,Agent 要停止自動核准流程。

六、Offline 驗證結果(7/1 - 7/20,共 20 天)
用歷史的 356 件案件跑離線測試,Agent 都有產生審核建議,沒有漏跑的案件,
任務完成率是 100%。跟廠務專家覆核比對後,審核結論一致率是 91%,超過原本訂的 85% 門檻。
